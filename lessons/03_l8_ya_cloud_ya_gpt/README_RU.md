# NER-пайплайн: извлечение именованных сущностей с помощью YandexGPT

Домашнее задание: извлечение именованных сущностей из договоров об оказании услуг
с помощью API YandexGPT.

Пайплайн извлекает 4 типа сущностей из юридических документов на русском языке:
- **parties** — стороны договора: название, ИНН, КПП
- **dates** — дата подписания и сроки действия
- **amounts** — суммы с указанием валюты
- **obligations** — сроки обязательств


## Задание

Разработать систему NER с помощью API YandexGPT, которая автоматически извлекает
реквизиты, даты, суммы и сроки из юридических документов и возвращает их в
структурированном виде (JSON).

Критерии приёмки:
- Настроено корректное взаимодействие с API YandexGPT
- Промпты успешно извлекают все 4 типа сущностей
- Результат представлен в машиночитаемом виде (JSON)

## Промпты

### Системный промпт

```
Ты — точный AI-помощник юриста.
Извлекай факты только из текста, без домыслов.
Если данных нет — верни null. Никогда не придумывай.
```

Ключевые решения при проектировании:

- `temperature=0.01` — минимальная креативность для структурных данных
- Few-shot пример JSON в пользовательском промпте — фиксирует схему вывода
- Явная инструкция `null` — предотвращает галлюцинации при отсутствии данных


### Пользовательский промпт (шаблон)

Извлекает `parties`, `dates`, `amounts`, `obligations` из текста договора
с примером JSON. Полный шаблон в `src/ner_client.py`.

## Тестовые документы

| Файл | Тип | Граничный случай |
| :-- | :-- | :-- |
| doc_01_consulting.txt | Консультационные услуги | Полный набор сущностей |
| doc_02_it_support.txt | IT-поддержка | Ежемесячная + годовая сумма |
| doc_03_cleaning.txt | Клининг | Длинный документ, truncation |
| doc_04_marketing.txt | Маркетинг | ИП без КПП |
| doc_05_accounting.txt | Бухгалтерия | Переменная стоимость |
| doc_06_nda_no_amount.txt | NDA | Нет суммы — тест на галлюцинацию |
| doc_07_training_ocr_noise.txt | Обучение | OCR-шум: "О" вместо "0", опечатки |

## Стратегии обработки текста

Модуль `chunker.py` поддерживает две стратегии (параметр `STRATEGY` в `ner_pipeline.py`):

- `truncate` — берёт начало и конец документа, пропускает середину; быстро, подходит для большинства договоров, где реквизиты идут в начале
- `chunk` — делит по абзацам, обрабатывает каждый чанк, объединяет результаты через `merge_chunks()`

Лимит токенов: ~4500 символов (~3500 токенов) с запасом на промпт и ответ.

## Выводы

### Плюсы LLM-подхода (YandexGPT)

1. Нулевой порог входа: не нужно размечать данные, обучать модель, писать правила.
Достаточно промпта — работает из коробки на любых типах документов.
2. Извлекает сущности, недоступные классическим моделям:
ИНН, КПП, р/с, сроки обязательств — типов, которых нет в Natasha и spaCy.
3. Понимает контекст: различает "Исполнитель" и "Заказчик", корректно
привязывает реквизиты к стороне договора.
4. Устойчив к OCR-шуму: понимает "ИНН 63О1234567" (О вместо 0),
опечатки в названиях — классические модели на таком деградируют.
5. Гибкость через промпт: новый тип сущности = правка промпта,
а не переобучение модели.

### Минусы LLM-подхода

1. Галлюцинации: в тестовом наборе не обнаружены, но ожидаемы на более слабых
моделях и структурно сложных документах при масштабировании.
2. Нестабильный JSON: в тестовом наборе не обнаружен, но вероятен на больших
объёмах и сложных документах исходя из общего поведения LLM.
3. Скорость и стоимость: 1-3 с на документ vs менее 200 мс у Natasha.
При 10 000 документов/день — значимые задержки и расходы на токены.
4. Зависимость от облака: облачная модель показала себя лучше, чем llama3
через Ollama с CPU-инференсом на ноутбуке со встроенной графикой.
5. Непредсказуемость: два одинаковых запроса могут дать разный результат
(решается temperature=0.01, но не полностью).

### Рекомендация для production

Оптимальный подход — гибридный:

**Классика (Natasha + regex)**

- ИНН, КПП, р/с — детерминировано, мгновенно, бесплатно
- ФИО, организации, даты — быстро, офлайн, без токенов
- Использовать как слой валидации вывода LLM

**LLM (YandexGPT)**

- Сроки обязательств, контекст сторон, OCR-зашумлённые документы
- Документы с нелинейной структурой (реквизиты не в порядке линейного чтения)

Чистый LLM-подход оправдан как прототип или при наличии человека для валидации.
В high-load production Natasha+regex надёжнее и дешевле для стандартных реквизитов
из заранее валидированных документов с известной структурой.
